# Release v1.9.0 - Schema-Level Classification Commands

**Release Date:** February 27, 2026  
**Status:** ✅ Production Ready  
**Previous Release:** [v1.8.5](v1.8.5.md) (February 20, 2026)

## 🎯 Overview

v1.9.0 introduces three new commands for managing **schema-level (column) classifications** in Microsoft Purview — filling a gap where `read-classifications` only returned entity-level classifications, not column-level ones visible in the Purview UI under "Classifications de schéma". This release also ships new utility scripts for copying data products and entities across collections, plus a comprehensive test suite.

### Key Statistics

- **3 New Commands:** `read-schema-classifications`, `add-schema-classification`, `remove-schema-classification`
- **2 New Scripts:** `copy_data_products.py`, enhanced `copy_entities_to_collection.py`
- **31 New Tests:** Full coverage for schema classification commands
- **Bug Fix:** Resolved `list` builtin shadowing issue in `entity.py` causing silent failures

---

## ✨ What's New

### 🏷️ Schema-Level Classification Commands

Microsoft Purview distinguishes between **entity-level classifications** (red badges) and **schema/column-level classifications** (purple badges). The existing `read-classifications` command only returned entity-level data. Three new commands address column-level classifications:

#### `pvw entity read-schema-classifications`

Fetches all column-level classifications for a table entity, displaying them in a Rich table or as JSON.

```powershell
# Table output (default)
pvw entity read-schema-classifications --guid <table-guid>

# JSON output
pvw entity read-schema-classifications --guid <table-guid> --output json
```

**Sample output:**

#### `pvw entity add-schema-classification`

Applies one or more classifications to columns. Supports three targeting modes:

```powershell
# Direct: target a column by its GUID
pvw entity add-schema-classification \
  --column-guid <col-guid> \
  --classification-name "Canada Social Insurance Number"

# By name: look up column by display name within a parent table
pvw entity add-schema-classification \
  --guid <table-guid> \
  --column-name NAS \
  --classification-name "Canada Social Insurance Number" \
  --classification-name "Disponibilite eleve"

# All columns at once
pvw entity add-schema-classification \
  --guid <table-guid> \
  --all-columns \
  --classification-name "Integrite eleve"

# Dry-run preview (no API calls)
pvw entity add-schema-classification \
  --guid <table-guid> \
  --all-columns \
  --classification-name "Test" \
  --dry-run
```

#### `pvw entity remove-schema-classification`

Removes one or more classifications from columns. Same three targeting modes as `add-schema-classification`. Gracefully skips columns that don't have the specified classification (no error).

```powershell
# Remove from a specific column by name
pvw entity remove-schema-classification \
  --guid <table-guid> \
  --column-name NAS \
  --classification-name "Canada Social Insurance Number"

# Remove from all columns (skips those without the classification)
pvw entity remove-schema-classification \
  --guid <table-guid> \
  --all-columns \
  --classification-name "Integrite eleve"
```

### 🛠️ New Scripts

#### `scripts/copy_data_products.py`

Copies `DataProduct` entities between Purview accounts or collections, preserving classifications, labels, contacts, and custom attributes.

```powershell
python scripts/copy_data_products.py \
  --source-endpoint https://source.purview.azure.com \
  --target-endpoint https://target.purview.azure.com \
  --collection TargetCollection
```

#### `scripts/copy_entities_to_collection.py` (enhanced)

Updated to handle classification assignment, metadata propagation, and improved error reporting when copying entities across collections.

## 🐛 Bug Fixes

### `list` Builtin Shadowed in `entity.py`

**Root cause:** A Click command named `list` (`@entity.command() def list(...)`) at line ~2856 of `entity.py` shadowed Python's builtin `list` for the entire module. This caused three distinct silent failures:

| Code | Failure |
|------|---------|
| `list(classification_names)` in `add_schema_classification` | Called Click Command → `sys.exit(2)` |
| `isinstance(col_classifs, list)` in `read_schema_classifications` | `TypeError` caught silently → 0/N columns counted |
| `isinstance(existing_resp, list)` in `remove_schema_classification` | `TypeError` → skip logic bypassed → unwanted deletions |

**Fix:** Added `_builtin_list = list` at module top (before the `def list` Click command) and replaced all affected calls.

### `add_classifications` Command Missing Definition

The `add_classifications` command body was left as orphaned floating code (no `def` or decorators) after a previous refactor. This caused a `NameError` at module import time. The proper decorator group and `def add_classifications(ctx, guid, payload_file):` signature have been restored.

## 🧪 Tests

New test file `tests/test_schema_classifications.py` with **31 tests** covering:

- Table and JSON output formats
- All three targeting modes (direct GUID, by column name, all columns)
- Case-insensitive column name matching
- Multi-classification payloads
- Dry-run mode
- API error capture (per-column, non-fatal)
- Graceful skip when classification not present
- Mock mode (`--mock` flag)
- Edge cases: entity not found, no columns, `referredEntities` fallback

All 31 tests pass against mocked API calls with no real Purview connectivity required.
