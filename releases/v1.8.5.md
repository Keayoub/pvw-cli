## 🎯 Overview

v1.8.5 adds native support for assigning entities directly to collections during creation, eliminating the need for separate move operations. This release also fixes a critical bug in the `move-to-collection` command and enhances entity example files with comprehensive metadata fields.

### Key Statistics

- **New Feature:** Collection assignment during entity creation
- **Bug Fixed:** AttributeError in `pvw entity move-to-collection` command
- **Enhanced Examples:** Updated entity JSON samples with contacts, labels, and metadata
- **API Compatibility:** Confirmed support for `collectionId` parameter in API 2023-09-01+

---

## ✨ What's New

### 🚀 Collection Assignment During Entity Creation

**Problem Solved:**
Previously, entities could only be created in the root collection, requiring a separate `move-to-collection` operation. This two-step process was inefficient for bulk operations.

**Solution:**
v1.8.5 adds `--collection-id` parameter to both `create` and `bulk-create` commands, allowing direct assignment to target collections.

**New Commands:**

```powershell
# Create single entity directly in a collection
pvw entity create --payload-file entity.json --collection-id yfwy2c

# Create multiple entities in bulk directly in a collection
pvw entity bulk-create --payload-file entities.json --collection-id yfwy2c
```

**Technical Implementation:**
- Added `--collection-id` CLI option to `entity create` and `entity bulk-create` commands
- Client methods now pass `collectionId` as query parameter to API endpoints
- Compatible with API versions 2023-09-01, 2024-03-01-preview, and later
- Follows Microsoft official documentation for bulk entity operations

**API Endpoint:**
```http
POST {endpoint}/datamap/api/atlas/v2/entity/bulk?api-version=2023-09-01&collectionId={collectionId}
```

**Benefits:**
- ✅ One-step entity creation with collection assignment
- ✅ Eliminates need for separate move operations
- ✅ Faster bulk imports
- ✅ Cleaner automation scripts

**Backward Compatibility:**
✅ **No Breaking Changes:** The `--collection-id` parameter is optional. If omitted, entities are created in the root collection (default behavior).

### 🐛 Bug Fixes

#### Fixed: AttributeError and API Compliance in `move-to-collection` Command

**Issue:**
```
[X] Error executing entity move-to-collection: 'Entity' object has no attribute 'entityMoveEntitiesToCollection'
```

**Root Causes:**
1. Method name mismatch between CLI command and client implementation
2. Non-compliant API usage: CLI expected `collectionId` in payload instead of query parameter

**Fix:**
1. Updated CLI command to call the correct method name `entityMoveToCollection()`
2. Refactored to match Microsoft API specification:
   - `collectionId` now passed as query parameter (required)
   - Payload contains only `entityGuids` array
   - Added `--collection-id` CLI parameter (required)
   - Added `--guids` CLI parameter for inline GUID list
   - `--payload-file` now optional (alternative to `--guids`)

**Files Changed:**
- `purviewcli/cli/entity.py` - Refactored move_to_collection command
- `purviewcli/client/_entity.py` - Updated entityMoveToCollection method
- `scripts/copy_entities_to_collection.py` - Updated to use new format
- `samples/json/entity/move_entities.json` - New example file

**New Usage (2 Methods):**

**Method 1: Direct GUIDs (Recommended for simple cases)**
```powershell
# Move entities by GUIDs directly
pvw entity move-to-collection --collection-id yfwy2c --guids guid1,guid2,guid3
```

**Method 2: Payload File (For large lists)**
```powershell
# Move entities using JSON file
pvw entity move-to-collection --collection-id yfwy2c --payload-file move_entities.json
```

**New Payload Format:**
```json
{
  "entityGuids": ["guid1", "guid2", "guid3"]
}
```

**Note:** `collectionId` is NO LONGER in the payload - it's now a required CLI parameter.

**API Compliance:**
Now correctly implements Microsoft Purview API specification:
```http
POST {endpoint}/datamap/api/entity/moveTo?api-version=2024-03-01-preview&collectionId={collectionId}
Body: { "entityGuids": [...] }
```

### 📄 Enhanced Entity Example Files

**Problem Solved:**
Example entity JSON files had minimal fields, not reflecting real-world usage with contacts, labels, and metadata.

**Solution:**
Updated sample files with comprehensive fields based on Microsoft Purview API documentation.

**Enhanced Fields Added:**
- **`contacts`**: Owner and Expert contacts with GUIDs and info
- **`labels`**: Tags for entity categorization (e.g., "Production", "Sensitive")
- **`createdBy`** / **`updatedBy`**: User attribution for audit trail
- **`version`**: Entity version control
- **`owner`**, **`createTime`**, **`modifiedTime`**: Additional attributes

**Updated Files:**
- `samples/json/entity/entity.json`
- `samples/json/entity/entities.json`

**Example (New Format):**
```json
{

    {

      "status": "ACTIVE",
      "attributes": {
        "qualifiedName": "mssql://sqlsvr.database.windows.net/sqldb/Schema/Table01",
        "name": "Table01",
        "description": "Table01 is a custom entity.",
        "objectType": "U ",
        "owner": "DataTeam",
        "createTime": 0,
        "modifiedTime": 0
      },
      "contacts": {
        "Owner": [
          {
            "id": "30435ff9-9b96-44af-a5a9-e05c8b1ae2df",
            "info": "Data Owner Contact"
          }
        ],
        "Expert": [
          {
            "id": "30435ff9-9b96-44af-a5a9-e05c8b1ae2df",
            "info": "Technical Expert Contact"
```

## 🔧 Technical Changes

### CLI Commands Modified

**`purviewcli/cli/entity.py`:**
- Added `--collection-id` parameter to `create()` function
- Added `--collection-id` parameter to `bulk_create()` function
- **Refactored `move_to_collection()` function:**
  - Added `--collection-id` parameter (required)
  - Added `--guids` parameter (optional, alternative to --payload-file)
  - Made `--payload-file` optional (alternative to --guids)
  - Fixed method call from `entityMoveEntitiesToCollection()` to `entityMoveToCollection()`
- Added conditional parameter passing for collection ID

### Client Methods Updated

**`purviewcli/client/_entity.py`:**
- **`entityCreateOrUpdate()`**: Now accepts `--collectionId` in args and passes to API params
- **`entityBulkCreateOrUpdate()`**: Now accepts `--collectionId` in args and passes to API params
- **`entityMoveToCollection()`**: 
  - Now requires `--collectionId` in args (passed as query parameter)
  - Accepts either `--payloadFile` or `--guids` for entity list
  - Constructs payload from comma-separated GUIDs when using `--guids`
  - Validates required parameters

### Scripts Updated

**`scripts/copy_entities_to_collection.py`:**
- Updated to pass `collectionId` as separate parameter instead of in payload
- Payload now contains only `entityGuids` array

### Sample Files Added

**`samples/json/entity/move_entities.json`:**
- New example file showing correct payload format (entityGuids only)

**Implementation Pattern:**
```python
self.params = get_api_version_params("datamap")
if "--collectionId" in args:
    self.params["collectionId"] = args["--collectionId"]
```

## 🔄 Migration Guide

### For Users Coming from v1.8.3 or Earlier

**⚠️ Breaking Change for `move-to-collection`:**
If you use `pvw entity move-to-collection`, you MUST update your commands:

**Before (v1.8.3):**
```powershell
# Old format: collectionId in payload
pvw entity move-to-collection --payload-file move.json

# move.json contained:
{
  "entityGuids": ["guid1"],
  "collectionId": "yfwy2c"
}
```

**After (v1.8.5):**
```powershell
# New format: collectionId as CLI parameter
pvw entity move-to-collection --collection-id yfwy2c --payload-file move.json

# OR use direct GUIDs (simpler)
pvw entity move-to-collection --collection-id yfwy2c --guids guid1,guid2

# move.json now contains ONLY:
{
  "entityGuids": ["guid1"]
}
```

**Other Commands:**
✅ **No Action Required** for `create`, `bulk-create`, and other commands. These remain backward compatible.

**Optional Enhancement:**
Replace two-step create + move workflows with single-step create:

**Before (v1.8.3):**
```powershell
# Step 1: Create entities (in root collection)
pvw entity bulk-create --payload-file entities.json

# Step 2: Move to target collection
pvw entity move-to-collection --payload-file move_payload.json
```

**After (v1.8.5):**
```powershell
# Single step: Create directly in target collection
pvw entity bulk-create --payload-file entities.json --collection-id yfwy2c
```

## ⚠️ Known Limitations

1. **Collection ID Format:**
   - Must be a valid collection identifier (e.g., 'yfwy2c')
   - Collection must exist before entity creation
   - User must have write permissions on target collection

2. **Contact GUIDs:**
   - Contact IDs must be valid Entra Object IDs (GUIDs)
   - Email addresses are NOT accepted for contacts
   - Maximum 20 contacts per contact type

3. **Business Attributes:**
   - Default behavior is 'merge' (not yet exposed in CLI)
   - Future release will add `--business-attribute-behavior` option

## 📝 Breaking Changes

**⚠️ Breaking Change for `move-to-collection` command:**

The `pvw entity move-to-collection` command now requires `--collection-id` as a CLI parameter instead of including it in the JSON payload. This change aligns with Microsoft Purview API specification.

**Migration Required:**
```powershell
# OLD (v1.8.3 and earlier) - NO LONGER WORKS
pvw entity move-to-collection --payload-file move.json
# where move.json had: {"entityGuids": [...], "collectionId": "xxx"}

# NEW (v1.8.5+) - REQUIRED FORMAT
pvw entity move-to-collection --collection-id yfwy2c --payload-file move.json
# where move.json has: {"entityGuids": [...]}

# OR use simpler format with inline GUIDs
pvw entity move-to-collection --collection-id yfwy2c --guids guid1,guid2
```

**All Other Commands:**
✅ Backward compatible - no changes required for `create`, `bulk-create`, and other commands.
